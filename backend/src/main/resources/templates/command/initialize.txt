创建或更新当前项目的 `LabexAgent.md` 规则文档。

目标是创建一个紧凑的指令文件，帮助未来的 LabexAgent 会话避免错误并快速上手。每一行都应该回答："如果没有帮助，Agent 可能会错过这个吗？" 如果不是，就删除它。

用户提供的关注点或约束（请遵守这些）：
$ARGUMENTS

## 如何调查

优先读取最高价值的源文件：
- `README*`、根清单、工作区配置、锁文件
- 构建、测试、lint、格式化、类型检查和代码生成配置
- CI 工作流和预提交/任务运行器配置
- 现有指令文件（`LabexAgent.md`、`CLAUDE.md`、`.cursor/rules/`、`.cursorrules`）
- 项目本地配置如 `package.json`、`pom.xml`、`requirements.txt`

如果在阅读配置和文档后架构仍不清楚，检查少量代表性代码文件以找到真正的入口点、包边界和执行流。优先阅读解释系统如何连接的文件，而不是随机的叶子文件。

可执行的真实来源优先于文档。如果文档与配置或脚本冲突，信任可执行源，只保留你能验证的内容。

## 提取什么

寻找在本仓库中工作的最高信号事实：
- 确切的开发者命令，特别是不明显的命令
- 如何运行单个测试、单个包或聚焦的验证步骤
- 命令顺序重要时，如 `lint -> typecheck -> test`
- 单体仓库或多包边界、主要目录的所有权、真正的应用/库入口点
- 框架或工具链怪癖：生成的代码、迁移、代码生成、构建产物、特殊环境加载、开发服务器、基础设施部署流程
- 与默认值不同的仓库特定风格或工作流约定
- 测试怪癖：固件、集成测试前提、快照工作流、所需服务、不稳定或昂贵的套件
- 值得保留的现有指令文件中的重要约束

好的 `LabexAgent.md` 内容通常是通过阅读多个文件推断出来的来之不易的上下文。

## 问题

只有在仓库无法回答重要问题时才询问用户。使用 `question` 工具最多一次短批次。

好的问题：
- 未文档化的团队约定
- 分支/PR/发布期望
- 缺少的设置或测试前提（已知但未写下来）

不要问仓库已经清楚的任何事情。

## 写作规则

只包含高信号、仓库特定的指导，如：
- Agent 可能会猜错的确切命令和快捷方式
- 从文件名不明显的架构注释
- 与语言或框架默认值不同的约定
- 设置要求、环境怪癖和操作陷阱
- 重要的现有指令源的引用

排除：
- 通用软件建议
- 长教程或详尽的文件树
- 明显的语言约定
- 推测性声明或你无法验证的任何内容
- 最好存储在通过 `opencode.json` `instructions` 引用的另一个文件中的内容

有疑问时，省略。

优先使用短段落和要点。如果仓库简单，保持文件简单。如果仓库大，总结实际改变 Agent 工作方式的少数结构事实。

如果 `LabexAgent.md` 已存在于 `${path}`，就地改进而不是盲目重写。保留已验证的有用指导，删除绒毛或过时的声明，并与当前代码库协调。
